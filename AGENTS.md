# Repository Guidelines

## Project Structure & Module Organization

GardenLog is an Astro 5 static site for documenting a personal plant collection. Application code lives in `src/`: reusable page pieces are in `src/components/`, the shared shell is `src/layouts/Layout.astro`, and routes are in `src/pages/`. Plant detail pages are generated by `src/pages/plants/[slug].astro` from `src/data/plants.json`. Keep plant photos in `public/images/` and refer to them with root-relative paths such as `/images/cilantro-1.jpeg`. `src/pages/*.html` contains older static pages; prefer the Astro routes for new work. Global and page-specific CSS is in `src/styles/`.

## Build, Test, and Development Commands

Use pnpm (the lockfile is `pnpm-lock.yaml`):

```bash
pnpm install       # install dependencies
pnpm dev           # run the local Astro server at http://localhost:4321
pnpm build         # type-check and generate the production site in dist/
pnpm preview       # serve the most recent production build
pnpm astro check   # run Astro/TypeScript validation directly
```

There is currently no automated test or lint script. Run `pnpm build` before opening a pull request; use `pnpm preview` when a visual change needs a production-like check.

## Coding Style & Naming Conventions

Follow the surrounding code: Astro and TypeScript blocks use tabs, while JSON and CSS use four spaces. Keep components PascalCase (`PlantCollection.astro`), route and asset names lowercase/kebab-case, and plant `slug` values URL-safe lowercase kebab-case (for example, `tomate-cherry`). Use the `@/` alias for imports from `src/`. Preserve the existing Spanish user-facing copy and add meaningful `alt` text to every image. Match existing CSS class naming (for example, `plants-grid` and `plant-card`) and keep styles responsive and accessible, including visible focus states and reduced-motion support.

## Plant Data & Content Changes

When adding a plant, add one complete object to `src/data/plants.json`, ensure its `slug` is unique, and provide every image referenced by `image` or `gallery` under `public/images/`. Check the generated `/plants/<slug>` page locally and verify that care details, problems, and tips render when optional fields differ by plant.

## Commit & Pull Request Guidelines

Recent history uses concise Conventional Commit-style subjects, such as `feat: add Hero and PlantCollection components` and `refactor: rewrite README`. Use `feat:`, `fix:`, `docs:`, `refactor:`, or `chore:` with an imperative summary. Keep pull requests focused; describe the visible or data change, link related issues when applicable, include screenshots for UI changes, and note the validation commands you ran.
